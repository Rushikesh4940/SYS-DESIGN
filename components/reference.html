CAP Theorem — The fundamental impossibilityFoundation
Eric Brewer, 2000
▼

In any distributed system, network partitions (P) are a fact of life — messages get lost, delayed, duplicated. The question becomes: when a partition occurs, do you preserve Consistency or Availability?

CP systems (HBase, Zookeeper, etcd): refuse to serve stale data. Return errors during partitions. Used for financial ledgers, configuration stores, leader election.

AP systems (Cassandra, CouchDB, DynamoDB): stay up and serve best-effort data. Resolve conflicts later with LWW, vector clocks, or CRDTs. Used for social feeds, carts, counters.

PACELC goes further: even without partitions, you face a latency (L) vs consistency (C) trade-off. Cassandra is PA/EL — it sacrifices consistency for both availability and latency.

// Real classifications:
// MySQL single node → CA (not partition tolerant)
// Cassandra         → AP (eventual consistency)
// HBase / Zookeeper → CP (strong consistency)
// DynamoDB          → AP by default, CP with option
// MongoDB           → CP (primary reads default)
// CockroachDB       → CP (distributed SQL)
PACELC Theorem — Beyond CAPAdvanced
Daniel Abadi, 2012
▼

CAP only deals with partition scenarios. PACELC adds: even when the system is running normally (Else), there's a trade-off between Latency and Consistency.

Full model: if Partition → (A or C) else (L or C). Every system makes both choices.

Dynamo-style systems (Cassandra, Riak) are PA/EL — sacrifice consistency for availability and latency.

VoltDB, Spanner are PC/EC — sacrifice availability and latency for strong consistency always.

// PACELC Classification:
// Cassandra    → PA/EL (tunable consistency)
// DynamoDB     → PA/EL (strong consistency opt-in)
// MongoDB      → PA/EC (replica set w/ write concern)
// Spanner      → PC/EC (TrueTime, external consistency)
// MySQL Cluster→ PC/EC
// PNUTS (Yahoo)→ PA/EL
Consistency Models — The full spectrumIntermediate
Strong → Weak
▼

Linearizability (Strict): Every read reflects the most recent write globally. Gold standard. Very expensive. Used in etcd, Zookeeper.

Sequential Consistency: All operations appear in the same sequential order across nodes, but not necessarily real-time order. Used in multi-processor memory models.

Causal Consistency: Operations that are causally related appear in the correct order. Concurrent operations can be seen in any order. Practical sweet spot.

Read Your Own Writes: After a write, that same client is guaranteed to see it. Simple, common requirement.

Eventual Consistency: Nodes will converge. No timing guarantees. Highest availability. Used in DNS, Cassandra default, S3.

// Linearizable: Spanner, etcd, ZooKeeper
// Sequential:   Multi-CPU RAM (TSO model)  
// Causal:       MongoDB (causal sessions)
// Read-my-writes: DynamoDB (same session)
// Monotonic read: Cassandra with consistency=ONE
// Eventual:     DNS, S3, Cassandra default
02 — Infrastructure

Scaling & Traffic

The building blocks of every high-scale system — load balancing, caching, CDNs, and the math behind scaling decisions.

Scaling Strategies — Vertical, Horizontal, and beyond
▼

Vertical scaling: More RAM, CPU, faster disk on one machine. Simple, no code change. Hard ceiling. Single point of failure.

Horizontal scaling: More machines, distribute load. Requires stateless services, distributed session, consistent hashing. Linear cost scaling.

Stateless vs Stateful: Stateless services (REST APIs, compute) scale horizontally trivially. Stateful services (databases, sessions) require sharding, replication, or external state stores.

Read replicas: Scale read traffic independently. Writes go to primary, reads distributed across replicas. 80% of workloads are read-heavy — this is high leverage.

Functional decomposition: Split a monolith by business domain into services. Each can scale independently. The cost is operational complexity.

// Scaling ladder (rough heuristics):
// 1K users   → 1 server + DB
// 10K users  → Read replica + cache
// 100K users → Horizontal API + CDN
// 1M users   → Microservices + sharding
// 10M+ users → Multi-region + async everywhere
Load Balancing — Algorithms, layers, and health checking
▼

L4 Load Balancing: Routes by IP/TCP. Faster, no content inspection. Used for raw TCP, UDP, non-HTTP. AWS NLB, HAProxy (TCP mode).

L7 Load Balancing: Routes by HTTP content — headers, cookies, path, query params. Enables path-based routing, A/B testing, sticky sessions. Nginx, AWS ALB.

Round Robin: Simple rotation. Works when servers are homogeneous.

Weighted Round Robin: Servers get traffic proportional to their capacity.

Least Connections: New request goes to server with fewest active connections. Better for variable-duration requests.

Consistent Hashing: Same client → same server. Essential for caching layers, session affinity. Virtual nodes handle unequal distribution.

Health checking: Active (periodic HTTP pings) or passive (track error rates). Remove unhealthy nodes automatically.

// Session stickiness options:
// 1. Client IP hash (inaccurate with NAT)
// 2. Cookie-based (LB injects cookie)
// 3. Application-level token (preferred)
// 4. Shared session store (Redis) ← best
Caching — Strategies, eviction, invalidationIntermediate
▼

Cache-aside (lazy loading): App checks cache → miss → load from DB → write to cache. Handles cache failures gracefully. Data only cached when needed. Most common.

Write-through: Write to cache and DB in same operation. Always consistent. Higher write latency. Wastes space if data is never read.

Write-back (write-behind): Write to cache, async flush to DB. Lowest latency. Risk of data loss on crash. Used for high-write counters.

Read-through: Cache handles fetching from DB on miss. Simpler app code. Used in CDNs, ORM-level caches.

Eviction policies: LRU (most common), LFU (frequency), TTL (time-based), FIFO. Redis supports LRU, LFU, TTL, and random.

Cache invalidation: The hardest problem. Strategies: TTL-based expiry, event-driven invalidation, cache versioning, two-phase invalidation.

Cache stampede (thundering herd): Many requests miss simultaneously on the same key. Solutions: mutex locking, probabilistic early expiration, background refresh.

// Cache sizing: target >90% hit rate
// Cold start: pre-warm from DB dumps
// Stampede: lock-based or staggered TTL
// Hotspot: local in-process L1 cache +
//          distributed Redis L2 cache
CDN & Edge Computing — Push vs Pull, global distribution
▼

CDN (Content Delivery Network): Geographically distributed cache servers (PoPs — Points of Presence). Serve static assets, cached API responses, and now even serverless functions at the edge.

Push CDN: You push content to CDN proactively. Good for static content that doesn't change often. Control over what's cached. Used by streaming platforms.

Pull CDN: CDN fetches from origin on first request, then caches. Simpler to manage. Potential cold start delay. Most common.

Edge Computing: Run code at CDN PoPs (Cloudflare Workers, AWS Lambda@Edge). Sub-5ms latency for auth, routing, A/B testing. Can't do heavy compute — stateless, limited CPU.

Cache purging: How to invalidate CDN cache? URL versioning (/v2/app.js), cache-control headers, purge API calls, surrogate keys (Fastly, Varnish).

// CDN use cases:
// Static assets     → JS, CSS, images
// API responses     → public, read-only, cacheable
// Video streaming   → chunked, edge-cached
// Auth at edge      → JWT verification, rate limiting
// Geographic routing→ serve closest region
Rate Limiting — Algorithms, distributed stateIntermediate
▼

Fixed Window Counter: Count requests per fixed time window. Simple. Allows 2x burst at window boundary.

Sliding Window Log: Store timestamps of all requests. Accurate but memory-intensive. O(n) per request.

Sliding Window Counter: Approximate sliding window using two fixed buckets. Good accuracy with O(1) space.

Token Bucket: Tokens added at fixed rate, consumed per request. Allows controlled bursting. AWS API Gateway default.

Leaky Bucket: Requests queued and processed at fixed rate. Smooths bursts. Adds latency under load. Good for outbound API calls.

Distributed rate limiting: Share counters in Redis with atomic INCR + EXPIRE. Or use Redis Cell (CL.THROTTLE) which implements GCRA. For very high throughput, use local approximation + centralized reconciliation.

// Redis token bucket (Lua for atomicity):
// KEYS[1] = rate_limit:{user_id}
// ARGV = {capacity, refill_rate, tokens_requested}
// Use EVALSHA for performance
// 
// Distributed: local counters per node
//   + periodic sync to Redis ~1s
//   = ~1% error acceptable for rate limiting

Storage & Persistence

Sharding & Partitioning — Splitting data across nodesAdvanced
▼

Horizontal sharding: Split rows across multiple DB instances. Each shard holds a subset of the data. Scale writes and storage independently.

Hash-based sharding: shard = hash(key) % N. Even distribution. Hard to add nodes (requires resharding). Consistent hashing solves the resharding problem — only K/N keys move when adding a node.

Range-based sharding: Shard by value range (A-M on shard 1, N-Z on shard 2). Natural for time-series data. Risk: hot shards if access is not uniform.

Directory-based sharding: A lookup service maps keys to shards. Maximum flexibility. Single point of failure (mitigate with replication + caching).

Problems with sharding: Cross-shard JOINs become application-level. Transactions across shards require 2PC (slow) or saga pattern. Resharding requires careful migration.

// Consistent hashing ring:
// - Place nodes + virtual nodes on ring
// - Key hashes to nearest clockwise node
// - Add node → only adjacent keys move
// - Virtual nodes → more even distribution
// 
// Used by: Cassandra, DynamoDB, Redis Cluster
Replication — Leader-follower, multi-leader, leaderlessAdvanced
▼

Leader-follower (master-replica): All writes go to leader. Followers replicate asynchronously. Reads can go to any. Simple, most common. MySQL, PostgreSQL default.

Synchronous vs async replication: Sync = durable, high latency. Async = fast, risk of data loss on leader failure. Semi-sync = at least one follower ACKs (MySQL semi-sync).

Multi-leader replication: Multiple leaders accept writes. Complex conflict resolution required. Use cases: multi-datacenter writes, offline-first apps (CouchDB).

Leaderless replication: Any node accepts writes. Quorum reads/writes (W + R > N) for consistency. Read repair + anti-entropy for convergence. Dynamo, Cassandra.

Replication lag: Followers can be seconds or minutes behind. Read-after-write inconsistency is real. Solutions: read from leader for writes, route by session, compare timestamps.

// Quorum math (Dynamo-style):
// N = replication factor (e.g., 3)
// W = write quorum  (e.g., 2)
// R = read quorum   (e.g., 2)
// If W + R > N: strong consistency
// If W = 1, R = N: fast writes, slow reads
// If W = N, R = 1: fast reads, slow writes
Database Internals — B-trees, LSM trees, WAL, MVCCAdvanced
▼

B-tree (OLTP standard): Balanced tree, pages of 4–16KB. Good for reads and range queries. In-place updates. Random writes are slow on disk. Used by MySQL InnoDB, PostgreSQL.

LSM tree (Log-Structured Merge): All writes go to in-memory MemTable → flushed to immutable SSTables on disk → compaction merges SSTables. High write throughput. Read amplification mitigated by Bloom filters. Used by Cassandra, RocksDB, LevelDB.

WAL (Write-Ahead Log): Every change written to log before the actual write. Enables crash recovery and replication. PostgreSQL WAL, MySQL binlog/redo log.

MVCC (Multi-Version Concurrency Control): Each transaction sees a snapshot. No read-write conflicts. Multiple versions of rows maintained. Old versions garbage collected. PostgreSQL, MySQL InnoDB, CockroachDB.

// B-tree: read O(log n), write O(log n)
// LSM:    read O(log² n), write O(1) amortized
// 
// Read amplification: how many pages per read
// Write amplification: how many disk writes per write
// Space amplification: storage overhead
//
// B-tree: low read amp, high write amp
// LSM:    high read amp, low write amp
Indexing — B-tree, hash, composite, partial, covering
▼

B-tree index: Ordered. Supports range queries (>, <, BETWEEN, LIKE 'x%'). Default index type. O(log n) lookup.

Hash index: O(1) exact lookups. No range queries. Used internally by some engines, explicitly in PostgreSQL (hash index type).

Composite index: Index on multiple columns (a, b, c). Usable for prefix queries: (a), (a, b), (a, b, c). NOT usable for just (b) or (c). Column order matters — put high-cardinality, high-equality columns first.

Partial index: Index only rows matching a WHERE clause. Smaller, faster. e.g., index on (email) WHERE status = 'active'.

Covering index: Index includes all columns the query needs. No table lookup required. Maximum read performance.

Write trade-off: Every index slows down writes (INSERT/UPDATE/DELETE must update all indexes). For write-heavy tables, fewer indexes.

// Query optimization checklist:
// 1. EXPLAIN ANALYZE the slow query
// 2. Add index on WHERE, JOIN, ORDER BY cols
// 3. Composite: equality cols first, range cols last
// 4. Covering: include SELECT cols in index
// 5. Avoid: functions on indexed columns
//    BAD:  WHERE YEAR(created_at) = 2024
//    GOOD: WHERE created_at BETWEEN ... AND ...
NewSQL & Distributed SQL — Spanner, CockroachDB, TiDBAdvanced
▼

NewSQL combines the SQL interface and ACID transactions of relational databases with the horizontal scalability of NoSQL. The holy grail.

Google Spanner: Globally distributed, externally consistent. Uses TrueTime (atomic clocks + GPS) to assign globally monotonic timestamps. Enables serializable transactions across data centers.

CockroachDB: Open-source Spanner-inspired. Uses HLC (Hybrid Logical Clocks). PostgreSQL-compatible wire protocol. Multi-region active-active.

TiDB: MySQL-compatible. Separates storage (TiKV) from compute (TiDB). OLTP + HTAP (hybrid transactional/analytical).

YugabyteDB: PostgreSQL + Cassandra APIs. Raft for consensus. Strong consistency by default.

// When to choose NewSQL:
// ✓ Need SQL + transactions + horizontal scale
// ✓ Multi-region writes required
// ✓ PostgreSQL/MySQL migration desired
// ✗ Ultra-low latency (adds ~10-50ms overhead)
// ✗ Extreme cost sensitivity (expensive to run)
// ✗ Simple key-value (overkill, use DynamoDB)
04 — Messaging & Patterns

Communication Patterns

Message queues, event streaming, sync vs async, and the architectural patterns that let services talk without coupling.

Message Queues vs Event Streaming — Kafka vs RabbitMQ
▼

Message Queue (RabbitMQ, SQS): Messages are tasks. Consumers compete. Once consumed, message deleted. Point-to-point or pub/sub. Good for job queues, task distribution.

Event Streaming (Kafka): Messages are events. Retained for configurable time. Any consumer can read at any offset independently. Consumer groups for parallel processing. Used for event sourcing, audit logs, stream processing.

Kafka key concepts: Topics → Partitions → Offsets. Producers write to partitions. Consumer groups share partitions (one partition per consumer in a group). Retention: time-based or size-based.

Exactly-once semantics: At-most-once (lose some), at-least-once (duplicates), exactly-once (hard, requires idempotent consumers + transactions). Kafka ≥0.11 supports EOS with producer transactions.

// Kafka throughput tuning:
// - Batch size: 16KB default, tune up for throughput
// - Compression: snappy or lz4 for network efficiency
// - Partitions: parallelism = min(partitions, consumers)
// - Replication factor: 3 for production
// - acks=all + min.insync.replicas=2 for durability
Saga Pattern — Distributed transactions without 2PCAdvanced
▼

When a transaction spans multiple services, you can't use a single ACID transaction. The Saga pattern breaks it into a sequence of local transactions with compensating transactions on failure.

Choreography-based: Each service publishes events. Other services react. No central coordinator. Decoupled. Hard to trace, risk of cyclic dependencies.

Orchestration-based: A central orchestrator (saga) tells services what to do step by step. Easier to reason about, single place to trace failure. Temporal.io, AWS Step Functions.

Compensating transactions: Each step must have a rollback. "Cancel reservation" compensates "reserve seat". Compensations must be idempotent.

// Order saga example:
// 1. Create order       → compensate: cancel order
// 2. Reserve inventory  → compensate: release stock
// 3. Charge payment     → compensate: issue refund
// 4. Schedule delivery  → compensate: cancel delivery
//
// If step 3 fails → run 2C, 1C in reverse order
CQRS & Event Sourcing — Separating reads from writesAdvanced
▼

CQRS (Command Query Responsibility Segregation): Separate models for reading and writing. Write side handles commands, updates the write model, publishes events. Read side consumes events, maintains optimized read models (projections) for each use case.

Benefits: Scale reads and writes independently. Read models optimized per query. Avoids complex JOINs on the write side.

Event Sourcing: Never store current state. Store the sequence of events that led to it. Current state = replay of events. Complete audit trail. Time travel. But: eventual consistency, complex queries, storage growth.

Projection (read model): A derived view rebuilt by replaying events. Can be rebuilt at any time. Supports multiple views from the same event log.

// Event Sourcing example (bank account):
// Events: Opened, Deposited(100), Withdrawn(30)
// Current state = sum of all events = 70
//
// Benefits: audit log, time travel debugging,
//           rebuild any view from scratch
// Drawbacks: querying current state = replay all
//            events (mitigate with snapshots)
API Design — REST, GraphQL, gRPC, WebSockets
▼

REST: Stateless, resource-based, HTTP verbs. Simple. Over-fetching and under-fetching issues. Good for public APIs, CRUD services.

GraphQL: Client specifies exactly what data it needs. Solves over/under-fetching. Complex server-side resolvers. N+1 query problem (mitigate with DataLoader). Good for complex data graphs, BFF pattern.

gRPC: Binary protocol (Protocol Buffers), strongly typed, HTTP/2. 5-10x faster than REST. Bidirectional streaming. Good for internal service-to-service, mobile clients where bandwidth matters.

WebSockets: Full-duplex persistent connection. Real-time bidirectional. Good for chat, live collaboration, gaming. Scale with sticky sessions or pub/sub fanout.

Server-Sent Events (SSE): Unidirectional server → client stream over HTTP. Simpler than WebSockets for one-way push (notifications, feeds).

// API versioning: URL /v2/ vs header vs content-type

Circuit Breaker Pattern — Preventing cascade failuresIntermediate
▼

When a downstream service is failing, continuing to send requests doesn't help — it makes things worse by exhausting threads and increasing load on a struggling system. The circuit breaker acts like an electrical breaker: open the circuit when failures exceed a threshold.

Closed state: Normal operation. Requests pass through. Track failure rate.

Open state: Failure threshold exceeded. Requests fail immediately (or use fallback). No calls to downstream.

Half-open state: After a timeout, allow one probe request. If successful → close. If failed → reopen.

// Hystrix / Resilience4j config:
// - failure threshold: 50% in 10s window
// - wait in open state: 60s
// - half-open: 5 probe calls
// - thread pool isolation: separate pool per dep
// - fallback: return cached data or default

08 — Advanced

Senior & Staff Level Topics

Consensus algorithms, CRDTs, multi-region architectures, and the concepts that differentiate senior from staff-level thinking.

Consensus
Raft Algorithm
Leader election, log replication, safety. Used by etcd, CockroachDB, TiDB. Simpler than Paxos but equivalent. Leader receives writes, replicates to followers, commits when quorum ACKs.
Must understand for: distributed DB design, service coordination questions
Conflict Resolution
CRDTs
Conflict-free Replicated Data Types — data structures that merge automatically without conflict. G-Counter, PN-Counter, LWW-Element-Set, OR-Set. Used by Redis, Riak, collaborative editors.
Must understand for: real-time collaboration, offline-first, distributed counters
Clocks
Vector Clocks & HLC
Vector clocks track causality: each node maintains a vector of logical clocks. HLC (Hybrid Logical Clock) combines physical and logical time. Used for conflict detection, causal ordering.
Must understand for: multi-master, conflict detection, event ordering
Global Scale
Multi-Region Active-Active
Multiple regions serve writes simultaneously. Requires conflict resolution (last-write-wins, CRDTs, application logic). Geo-routing sends users to nearest region. Async cross-region replication.
Must understand for: global SaaS, latency-sensitive apps, disaster recovery
Data Processing
Lambda & Kappa Architecture
Lambda: batch layer (Hadoop/Spark) + speed layer (Flink/Storm) + serving layer. Kappa: replace batch with replayable stream (Kafka). Kappa is simpler; Lambda has stronger batch guarantees.
Must understand for: analytics pipelines, real-time + historical data
Observability
Metrics, Logs, Traces
The three pillars: Metrics (Prometheus, time-series counters/gauges/histograms), Logs (structured JSON, ELK/Loki), Distributed Tracing (OpenTelemetry, Jaeger, correlation IDs). Essential for debugging distributed systems.
Must understand for: production operations, debugging latency in microservices
Microservices
Service Mesh
Sidecar proxies (Envoy) handle cross-cutting concerns: mTLS, retries, circuit breaking, observability, traffic splitting. Control plane (Istio, Linkerd) manages config. Removes concerns from app code.
Must understand for: large microservice deployments, zero-trust networking
Storage
Data Lake & Lakehouse
Data Lake: raw data in object storage (S3/GCS). Data Warehouse: structured, optimized for analytics (Redshift, BigQuery). Lakehouse (Delta Lake, Iceberg): ACID transactions on lake format = best of both.
Must understand for: analytics systems, data pipeline design
Search
Inverted Index & Relevance
Elasticsearch/Lucene: tokenize → stem → inverted index (term → doc list). TF-IDF and BM25 for relevance scoring. Sharding by doc range. Refresh interval: near-real-time 1s. Used for full-text, log search.
Must understand for: search feature, log analysis, Elasticsearch design
Two-Phase Commit vs Saga — Distributed transaction strategiesAdvanced
▼

2PC (Two-Phase Commit): Phase 1 (Prepare): coordinator asks all participants to prepare and lock resources. Phase 2 (Commit/Abort): if all prepared → commit; else → abort all.

Problems with 2PC: Blocking protocol — if coordinator crashes after prepare but before commit, all participants are locked indefinitely. Severely limits availability and throughput.

3PC: Adds a pre-commit phase to reduce blocking. Still not partition-tolerant. Rarely used in practice.

Saga vs 2PC: Sagas are non-blocking and partition-tolerant but require compensating transactions and accept eventual consistency. 2PC is strongly consistent but blocking. For microservices: Saga almost always preferred.

// When to use 2PC (rare):
// ✓ Same datacenter, known participants
// ✓ XA transactions across heterogeneous DBs
// ✓ Short transactions (<100ms)
// ✗ Cross-region (high latency locks)
// ✗ Microservices (too many services to coordinate)
//
// When to use Saga (common):
// ✓ Microservices with independent DBs
// ✓ Long-running business processes
// ✓ Need availability over strong consistency
Raft Consensus Algorithm — Leader election and log replicationAdvanced
▼

Raft guarantees that a cluster of N nodes can tolerate floor((N-1)/2) failures while remaining strongly consistent. A 3-node cluster tolerates 1 failure; a 5-node cluster tolerates 2.

Leader election: Nodes start as followers. Election timeout (150-300ms random). Candidate increments term, requests votes. Winner becomes leader, sends heartbeats to prevent re-election.

Log replication: Client writes go to leader. Leader appends to log, sends AppendEntries to followers. Once majority ACK → entry committed → leader responds to client → followers apply.

Safety: Only nodes with up-to-date log can win election. Ensures committed entries never lost.

Log compaction: Snapshot state + truncate log to prevent unbounded growth. Nodes can catch up via snapshot transfer.

// Raft guarantees:
// - Election safety: ≤1 leader per term
// - Leader append-only: never overwrites
// - Log matching: same index+term → same entry
// - Leader completeness: committed entries present
// - State machine safety: apply same commands same order
//
// Used by: etcd, CockroachDB, TiKV, Consul
09 — Interview Strategy

The Framework That Wins

A structured, repeatable approach for every system design interview — from SDE-2 to Principal level.

1
Clarify Requirements (5 min)
Never design without asking. Functional requirements (what it does), non-functional (scale, latency, availability). Ask about DAU, read/write ratio, geography, SLA.
→ "100M DAU, read-heavy 100:1, 99.99% uptime, global users?"
2
Capacity Estimation (5 min)
Back-of-envelope. QPS, storage, bandwidth. This drives your database choice, caching strategy, and server count. Know your powers of 10.
→ "100M × 10 req/day = 11.5K RPS, 100:1 ratio = 115 write/s"
3
API Design (5 min)
Define the public interface. REST endpoints or gRPC. Input/output schemas. Pagination strategy. This forces you to think about the data model before the architecture.
→ "POST /tweets, GET /feed?user_id&cursor, DELETE /tweets/:id"
4
High-Level Design (10 min)
Draw the block diagram. Client → CDN → LB → services → DB → cache. Walk through the happy path. Make explicit DB schema and key decisions. Get interviewer buy-in before diving deep.
→ Draw boxes + arrows, name every component, show data flow
5
Deep Dive (15 min)
Pick the hardest/most interesting part and go deep. Or follow the interviewer's lead. DB schema, scaling a bottleneck, failure handling, algorithmic core. This is where senior candidates shine.
→ "Let's talk about how you'd handle the celebrity user problem"
6
Trade-offs & Bottlenecks (5 min)
Proactively identify what breaks at 10x scale. What's the single point of failure? What would you do differently with infinite time? Strong candidates think critically about their own design.
→ "The fanout approach breaks for celebrities — I'd use a hybrid model"
